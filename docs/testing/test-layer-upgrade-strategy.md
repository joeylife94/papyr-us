# Test Layer 업그레이드 전략 (Layer 0~6)

## 1) 현재 상태 진단 요약

현재 저장소는 이미 `test:static`부터 `test:visual`까지 7개 레이어가 분리되어 있고,
`test:all`로 순차 실행되는 구조를 갖추고 있습니다. 특히 Layer 5/6은 Docker 기반
테스트 인프라 자동 기동/정리까지 포함하고 있어 운영 성숙도가 높은 편입니다.

다만 “실제 개발된 내용을 확실하게 검증”하고, “스크린샷만 봐도 동작을 이해”하려면
다음 세 가지 축을 더 강화해야 합니다.

1. **요구사항 추적성(Traceability)**: 기능 요구사항 ↔ 테스트 케이스 ↔ 증거(스크린샷/로그) 연결
2. **시나리오 밀도(Flow Depth)**: 단순 접근성/렌더 확인을 넘어 실제 사용자 작업 흐름 커버
3. **증거 품질(Evidence Quality)**: 결과물(스크린샷, trace, 영상, a11y 리포트)을 읽기 쉬운 형태로 정규화

---

## 2) 목표 품질 수준 정의 (Definition of Done for Test Layers)

각 레이어의 목적을 “실행 여부”가 아니라 “품질 보장 수준”으로 재정의합니다.

### Layer 0 · Static
- 목표: **빌드/배포 전에 결함 가능성을 1차 차단**
- 업그레이드 방향:
  - ESLint 규칙을 단순 문법에서 **안전/복잡도/테스트 품질 지표**까지 확장
  - PR에서 변경 파일 기준으로 경고 트렌드(증가/감소) 비교

### Layer 1 · Unit
- 목표: **핵심 함수 로직의 결정론 보장**
- 업그레이드 방향:
  - 경계값/반례(property-based testing 후보 포함) 강화
  - “함수 커버리지”보다 **분기 커버리지 + mutation score**를 KPI로 전환

### Layer 2 · Domain
- 목표: **비즈니스 불변식 보호**
- 업그레이드 방향:
  - 도메인 규칙을 “문장형 계약(예: always/never)”으로 정리 후 테스트에 1:1 매핑
  - 도메인 이벤트 순서(예: 생성→권한변경→조회) 불변식 검증 추가

### Layer 3 · Contract
- 목표: **FE/BE 인터페이스 파손 방지**
- 업그레이드 방향:
  - 스키마 검사 + **semver 정책 검사**(breaking/non-breaking 자동 분류)
  - fixture를 단순 정상 케이스뿐 아니라 **오류 응답 표준**까지 확장

### Layer 4 · Integration
- 목표: **실 인프라 연동 신뢰성 확보**
- 업그레이드 방향:
  - DB/Redis 장애 주입(지연, 연결 끊김, 재시도) 테스트 추가
  - 데이터 정합성(트랜잭션 롤백, 동시성 충돌) 검증 시나리오 강화

### Layer 5 · E2E
- 목표: **사용자 관점 핵심 업무 흐름 보장**
- 업그레이드 방향:
  - 현재 reachability 중심에서 **업무 시나리오 중심**으로 전환
  - Given-When-Then 기반 “행동 + 기대결과 + 증거 캡처 포인트” 명시

### Layer 6 · Visual
- 목표: **UX 회귀를 즉시 식별 가능한 형태로 가시화**
- 업그레이드 방향:
  - 페이지 단위 스냅샷에서 **상태(state) 단위 스냅샷**으로 확장
  - 시각 diff + a11y 결과 + 주요 DOM 상태를 하나의 리포트로 결합

## 3) 가장 중요한 UX 강화 전략 (핵심)

사용자가 원하는 수준은 “스크린샷만 봐도 실제 동작을 유추”할 수 있는 테스트입니다.
이를 위해 E2E/Visual을 아래처럼 재구성하는 것을 권장합니다.

### 3.1 시나리오를 `User Storyboard` 단위로 정의

각 시나리오를 5~9개의 화면 단계(step)로 고정합니다.

예시 (로그인 + 문서 작성 + 저장):
1. 로그인 페이지 진입
2. 자격 입력
3. 로그인 성공 후 대시보드
4. 문서 생성 모달 오픈
5. 제목/내용 입력
6. 저장 클릭
7. 저장 성공 토스트
8. 목록에 신규 문서 반영

각 단계마다 반드시 남길 것:
- 스크린샷 파일
- 핵심 assertion (예: 토스트 텍스트, 리스트 아이템 존재)
- 필요 시 trace/video 링크

### 3.2 스크린샷을 “디버그 아티팩트”가 아니라 “검증 산출물”로 취급

권장 규칙:
- 파일명에 시나리오/단계/상태 포함
  - 예: `login-flow__03-dashboard__success.png`
- 전역 공통 환경 고정
  - viewport, timezone, locale, colorScheme, animation disable
- 동적 요소 마스킹
  - timestamp, 랜덤 ID, 실시간 배지 등 불안정 영역

### 3.3 E2E에 `step screenshot` 표준 API 도입

Playwright `test.step()`마다 캡처하도록 공통 헬퍼를 둡니다.

권장 동작:
- step 시작 전/후 캡처 (필요 시)
- 실패 시 자동 최종 캡처
- 캡처와 assertion을 같은 블록에 두어 “왜 찍었는지” 명확화

### 3.4 Visual Layer를 “페이지 기준”에서 “상태 조합 기준”으로 확장

현재 `/`, `/login` 중심이라면 다음 상태 축을 추가합니다.
- 인증 상태: 비로그인 / 로그인
- 데이터 상태: empty / populated / error
- 디바이스 상태: desktop / tablet(최소)
- 테마 상태: light / dark (필요 시)

이렇게 하면 같은 페이지라도 UX 리스크가 큰 상태를 촘촘히 잡아낼 수 있습니다.

### 3.5 테스트 결과 리포트에 “스토리보드 인덱스” 생성

CI 아티팩트에서 아래를 한 번에 보이게 만듭니다.
- 시나리오명
- 단계별 썸네일
- 단계별 assertion 결과
- 실패 단계 하이라이트

즉, 리포트 첫 화면에서 “어디까지 정상 진행됐는지” 즉시 파악 가능해야 합니다.

## 4) 레이어별 구체 업그레이드 백로그

### Layer 0~3 (신뢰 기반 만들기)

1. **Test Impact Analysis**
   - 변경 파일을 기준으로 관련 레이어 우선 실행 (빠른 피드백)
2. **Schema Drift Alert**
   - Contract에서 필드 제거/타입 변경 시 breaking 경고 강화
3. **Domain Rule Catalog**
   - 도메인 규칙 문서화 + 테스트 ID 매핑

### Layer 4 (실행환경 리스크 흡수)

1. **회복성 시나리오**
   - DB 재연결/타임아웃/락 경합 테스트
2. **테스트 데이터 시딩 표준화**
   - 시나리오별 fixture seed 모듈화
3. **실패 분석성 강화**
   - 실패 시 쿼리 로그/서비스 로그 스냅샷 보관

### Layer 5~6 (UX 증거 중심)

1. **Critical User Journey 10개 선정**
   - 로그인, 검색, 문서 생성/수정, 댓글, 협업, 권한 변경 등
2. **단계별 스크린샷 의무화**
   - 각 journey 최소 5장 이상
3. **A11y 범위 확대**
   - critical만이 아니라 serious까지 경고 레벨로 수집
4. **시각 회귀 안정화**
   - 스냅샷 마스킹/폰트 고정/애니메이션 비활성화 표준 적용

## 5) 운영 규칙 제안 (품질 게이트)

### PR 게이트
- 필수 통과: Layer 0~4
- 변경 유형이 UI 관련일 때 필수 추가: Layer 5/6
- 신규 UX 기능 PR: “스토리보드 스크린샷” 없으면 머지 금지

### 릴리스 게이트
- 최근 7일 기준 flaky test 상위 목록 공개
- Layer 5/6 실패율이 임계치 초과 시 릴리스 보류

### KPI
- 결함 조기발견율(어느 레이어에서 발견됐는지)
- E2E mean time to diagnose (스크린샷/trace 도입 전후 비교)
- Visual false positive 비율

---

## 6) 4주 실행 로드맵 (실행 가능 버전)

### Week 1
- Critical Journey 10개 선정
- Journey별 step 정의 및 스크린샷 네이밍 규칙 확정
- E2E 공통 헬퍼(단계 캡처) 도입

### Week 2
- Journey 1~5 구현 + CI 아티팩트 정리
- Visual 상태 조합(인증/empty/populated/error) 일부 도입

### Week 3
- Journey 6~10 구현
- flaky 원인 상위 3개 제거 (비동기 대기, 테스트 데이터, 동적 UI)

### Week 4
- 리포트 스토리보드 인덱스화
- KPI 대시보드/운영 규칙 적용
- 품질 기준 재평가 및 다음 분기 백로그 수립

---

## 7) 바로 적용 가능한 실무 팁

1. **`waitForTimeout` 최소화**: 가능한 `locator`/`networkidle`/`event` 기반 대기 사용
2. **테스트 데이터 고정**: 날짜/랜덤/시간대 입력값 통제
3. **실패 시 증거 3종 세트**: screenshot + trace + video
4. **시나리오 문장화**: 테스트 코드 제목만 읽어도 비즈니스 흐름이 보이게 작성
5. **“무엇이 보이면 성공인가”를 명시**: UX assertion을 텍스트/역할(role)/상태로 구체화

---

## 8) 결론

현재 구조는 이미 좋은 출발점입니다. 다음 단계는 “테스트 개수 증가”가 아니라,
**사용자 시나리오 단위의 증거 품질**을 높이는 것입니다.

특히 UX 관점에서는 Layer 5/6을 결합해 “동작(E2E) + 화면(Visual) + 접근성(A11y)”을
한 번에 보여주는 리포트 체계를 구축하면, 스크린샷만으로도 제품 상태를 신뢰할 수 있는
수준에 도달할 수 있습니다.
